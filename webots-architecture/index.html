Problem Statement

— why this matters

The WEBots humanoid robot needs reliable distributed communications — but single-ended serial or WiFi links are vulnerable near PWM signals, motor drivers, and switching power supplies in a robot chassis.

Multiple ankle, knee, and subsystem motor controllers require deterministic, noise-resistant communication under motor-drive electrical noise. Meanwhile, higher-level robot behavior decisions live in a ROS 2 graph and need to integrate cleanly with the ESP32 firmware.

This project builds a dual-layer communications stack: CAN 2.0B for local control data and micro-ROS for ROS 2 graph integration — designed to scale from prototype modules to the electrical team's future custom PCB.

Technical Architecture

— two-layer communication stack
⚡

CAN 2.0B Layer

Low-level, noise-resistant bus for deterministic joystick telemetry between ESP32 nodes over MCP2515 controllers at 500 kbps. Fire-and-forget with application-level ACK.

Sender reads joystick X/Y ADC values, packs them into 8-byte CAN frames on ID 0x036, waits up to 500 ms for ACK ID 0x037, and retries up to 3 times. Loop runs at ~25 Hz.

🤖

micro-ROS Layer

High-level ROS 2 integration path via micro-ROS serial transport connecting the ESP32 to a full ROS 2 Jazzy agent. Bidirectional: ESP32 publishes topics and responds to LED control commands.

Publishers: /joystick_x_pub (50 ms), /goose_calls (400 ms). Subscriber: /led_control accepts "ON"/"OFF".

🔗

ACK Reliability

Application-level ACK mechanism gives command delivery visibility instead of fire-and-forget. The receiver decodes X/Y values, applies a 500-unit deadzone, sends an immediate ACK, and displays direction on an SSD1306 OLED.

The ACK timeout and retry strategy exposes delivery failures immediately — critical for integration with motor-controller feedback loops.

📦

Scalable for PCB

The communication stack is designed to port directly to the electrical team's future custom PCB. CAN 2.0B is deterministic, widely supported, and compatible with future MCP2551 or TJA1050 transceivers.

Next steps include a CAN transport layer that fragments ROS 2 messages larger than 8 bytes and reassembles them with per-frame ACKs. 

// CAN frame — joystick telemetry on ID 0x036
canMsg.can_id  = 0x036;
canMsg.can_dlc = 8;
memcpy(&canMsg.data[0], &VRX, sizeof(VRX));
memcpy(&canMsg.data[4], &VRY, sizeof(VRY));

// micro-ROS publisher initialization
rclc_publisher_init_default(
  &esp32_pub, &node,
  ROSIDL_GET_MSG_TYPE_SUPPORT(std_msgs, msg, Int32),
  "joystick_x_pub");

Design Challenges

— decisions and trade-offs

Electrical noise in robot control buses

Single-ended links fail near PWM/motor drivers. CAN differential signaling adds wiring complexity but directly targets noise immunity. Chose CAN 2.0B over MCP2515 for deterministic timing and future PCB compatibility.

Reliable local command delivery

Fire-and-forget CAN frames hide dropped commands. Adding application-level ACK costs bus time but exposes failures immediately. Implemented ACK ID 0x037 with 500 ms timeout and 3 retries.

Joystick drift and deadzone tuning

Raw ADC values fluctuate around center creating false direction changes. A larger deadzone improves stability but reduces sensitivity. Used a 500-unit deadzone as a practical prototype threshold.

CAN prototypes → ROS 2 integration

CAN is excellent for local control; higher-level behavior needs ROS graph integration. micro-ROS serial transport is simpler than custom CAN transport but depends on a host agent. micro-ROS node was built first, leaving custom CAN transport as the next step.

Status & Roadmap

— what's next

Working

  • CAN sender transmits joystick telemetry at 25 Hz
  • CAN receiver decodes telemetry, sends ACKs, drives OLED
  • micro-ROS node initializes with ROS 2 Jazzy agent
  • ESP32 publishes /joystick_x_pub and /goose_calls
  • ESP32 subscribes to /led_control

Limitations

  • Error logging not yet centralized
  • Deadzone tuned for prototype joystick, not motor feedback
  • micro-ROS service client prepared but not fully exercised
  • No ROS 2-over-CAN fragmentation/reassembly layer
Roadmap: Add structured error counters for CAN send failures and ACK timeouts. Complete the custom average4 service client path using generated ROSIDL types. Design a CAN transport layer that fragments ROS 2 messages larger than 8 bytes with per-frame ACKs. Validate against the electrical team's custom PCB and real servo/BLDC loads.
