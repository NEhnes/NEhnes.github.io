ESP32 WiFi RC Video Rover

Live video, joystick control, and real-time embedded trade-offs on a tiny ESP32-S3 platform.

A WiFi-controlled rover featuring an OV2640 camera stream, binary WebSocket control, adaptive frame skipping, active cooling, and a measured one-hour runtime on a 3S LiPo pack.

Problem statement

— real-time control under wireless constraints
Responsive teleoperation is a latency problem before it is a video problem. The rover streams live camera frames and receives motor commands over the same WiFi link, but joystick response cannot wait behind slow or dropped video frames. The project solves that by separating control from video, monitoring WebSocket buffer pressure, and degrading visual feedback before control quality collapses.

Technical approach

— asynchronous architecture and graceful degradation

Decoupled control loop

Motor commands arrive through binary WebSocket messages and update immediately. Video frames are captured and sent asynchronously, so a congested stream cannot block the latency-critical motor path.

Buffer-aware frame skipping

Before sending a JPEG frame, firmware checks WebSocket transmit capacity. If the buffer is full, the frame is skipped rather than stalling the rover.

DRAM-first frame buffers

The camera frame buffer is placed in DRAM with double buffering to reduce compression and transmission latency, trading scarce heap space for faster real-time response.

Thermal and power management

Continuous WiFi, JPEG encoding, and camera ISP work create thermal stress. The design uses a heat sink, 5V mini fan, temperature monitoring, and dynamic framerate throttling.

Design challenges

— constraints that shaped the system

Latency on bandwidth-constrained wireless

Sequential video and motor processing would make joystick response depend on camera frame timing. The solution uses asynchronous WebSocket handling so motor updates are never skipped.

WebSocket buffer exhaustion under congestion

Weak mobile links can fill the transmit buffer. The rover checks buffer capacity before sending frames and skips video rather than blocking control.

Memory architecture and latency trade-offs

DRAM frame buffers consume precious heap space but reduce frame access and compression latency compared with PSRAM.

Thermal stress in a small footprint

Continuous WiFi, encoding, and camera ISP work can push the ESP32 toward throttling. Active fan cooling and temperature-aware throttling preserve performance.
