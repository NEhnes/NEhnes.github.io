Problem Statement

— why this was built

Inspired by the nimble reconnaissance drones seen in Rainbow Six Siege, this project maps complex real-time constraints onto consumer-grade, ultra-small form factor hardware.

The core engineering hurdle is managing high-bandwidth, latency-sensitive video streaming alongside immediate motor control safety loops over a singular, volatile 2.4 GHz wireless link—all while operating under severe memory ceilings and high thermal dissipation requirements within a compact enclosure.

Overview

— scope and approach

Hardware Subsystems

Powered by a Seeed Studio XIAO ESP32-S3 (8MB Flash, 8MB PSRAM) paired with an OV2640 image sensor. Actuation is managed via a TB6612FNG H-bridge with integrated flyback protection driving twin JGA25-370 300 RPM DC gearmotors.

Software Ecosystem

Firmware hand-coded in C++ using the PlatformIO development environment on top of the Arduino hardware abstraction layer. Networking leverages a non-blocking AsyncTCP WebSocket layout hooked into a custom HTML5/JavaScript mobile interface.

Engineering Challenges

— deep-dives & implementations

CH 01 Real-Time Latency on Shared Wireless Pipelines

The Problem & Root Cause

A sequential processing pipeline meant that massive JPEG frame transmissions (~80-120 KB per frame) completely blocked incoming motor velocity updates. Wireless reception drops instantly ballooned motor reaction latencies, making control completely unsafe.

Engineering Impact

Decoupled motor control latency to <50ms independent of video payload strain by completely separating blocking I/O from critical routines.

Asynchronous Decoupled Firmware

CH 02 WebSocket Buffer Starvation Under Network Congestion

The Problem & Root Cause

When utilizing mobile hotspots or operating in congested spaces, the outbound network pipe narrows. Trying to force fixed-framerate streams down a saturated connection caused socket queue bottlenecks, crashing the connection entirely.

Engineering Impact

Maintained structural control over telemetry streams on volatile networks by implementing adaptive, buffer-aware frame-dropping loops.

Buffer-Aware Frame-Skip Execution

C++ 
void broadcastCameraFrame() {
  camera_fb_t *fb = esp_camera_fb_get();
  if (!fb) return;

  // Evaluate WebSocket TX buffer space before write allocation
  if (ws.availableForWriteAll()) {
    ws.binaryAll(fb->buf, fb->len);
  } else {
    // Gracefully drop frame if network congestion is detected
    Serial.println("Skipped frame: TX Buffer Full");
  }
  esp_camera_fb_return(fb);
}

CH 03 Memory Architecture & Latency Mappings

The Problem & Root Cause

The ESP32-S3 contains ultra-fast DRAM (~10ns access) and larger but slower external PSRAM (~50ns access). Storing frame buffers natively inside the slower PSRAM created severe data access overhead across millions of pixels, slowing down JPEG encoding steps.

Trade-off Evaluation

Sacrificed 40% of precious heap memory space within internal DRAM to accelerate JPEG compression passes, cutting total transmission delay by 10-15ms per frame.

DRAM-First Allocation Blueprint

C++ 
camera_config_t config;
// Pin buffer arrays explicitly to fast DRAM arrays
config.fb_location = CAMERA_FB_IN_DRAM;  
// Double-buffering yields parallel capture/send pipelines
config.fb_count = 2;

CH 04 Thermal Performance Constraints on Dense Footprints

The Problem & Root Cause

Simultaneous 2.4 GHz radio transmission and continuous hardware JPEG encoding inside a small enclosure causes the chip to output ~1W of localized heat. Left unchecked, silicon temperatures breached 80°C, causing thermal throttling and connection loss.

Engineering Impact

Stabilized continuous runtime temperatures within safe margins (65-75°C) by designing a custom active fan circuit and linking it with dynamic hardware throttling code loops.

Active Cooling Initialization

void setup() {
